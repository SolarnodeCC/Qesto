## Namespace Structure

```
src/locales/
  en/
    common.json       # Buttons, labels, errors shared across all pages
    session.json      # Session create/config/lifecycle strings
    vote.json         # Voter-facing strings
    presenter.json    # Presenter-facing strings
    admin.json        # Admin panel strings
  nl/
    (same structure)
  es/ de/ fr/
    (same structure)
```

**Rule**: A string belongs to the namespace of the page/component where it's first rendered. If used in 3+ namespaces, move to `common`.

## Translation Key Convention

```json
// Format: "component.context.action" or "component.label"
// Keys are camelCase dot-separated paths

// common.json
{
  "button.save": "Save",
  "button.cancel": "Cancel",
  "error.generic": "Something went wrong. Please try again.",
  "error.unauthorized": "You don't have permission to do this."
}

// session.json
{
  "config.title.label": "Session title",
  "config.anonymity.label": "Anonymity mode",
  "config.anonymity.options.anonymous": "Fully anonymous",
  "lifecycle.startButton": "Start session",
  "lifecycle.closeButton": "Close session"
}
```

**Never** use full sentences as keys — use semantic paths. Bad: `"Click here to start your session"`. Good: `"lifecycle.startButton"`.

## Language Detection Waterfall

```typescript
// Priority order (I18N-001)
1. URL param: ?lang=nl
2. localStorage: 'qesto_lang'
3. Browser: navigator.language (e.g. 'nl-NL' → 'nl')
4. CF Worker: Accept-Language header (server-side, prevents FOUC)
5. Fallback: 'en'
```

Server-side detection in `functions/api/[[route]].ts`:
```typescript
const lang = detectLanguage(c.req.header('Accept-Language')) ?? 'en'
c.header('X-Detected-Lang', lang)
```

## String Extraction Workflow

```bash
# 1. Extract all strings from src/ into en/ namespace files
npm run i18n:extract

# 2. Review generated keys — rename any auto-generated ones to semantic paths
# (formatjs generates hashes for inline JSX — rename before committing)

# 3. Validate all namespaces have complete keys
npm run i18n:validate
# Fails if any non-EN namespace has missing keys

# 4. Machine-translate missing keys (Workers AI — no Anthropic)
# For each missing key, use ai.ts to generate draft translation
# Mark draft translations with a comment for native reviewer: // AI draft

# 5. Native speaker review (NL required before Sprint 15; ES/DE/FR for Sprint 15)
```

## CI Validation

`npm run i18n:validate` must pass in CI — fails on:
- Missing key in any non-EN namespace (key exists in `en/` but not in target lang)
- Empty string value (`""`)
- Duplicate keys within a namespace

Missing keys at runtime: log to Workers Logs (`logError()`) + render EN fallback — never crash.

## Pluralisation

```typescript
// Use i18next pluralisation — never manual ternary
t('session.voterCount', { count: voterCount })
// en/session.json:
// "voterCount_one": "{{count}} voter",
// "voterCount_other": "{{count}} voters"

// German: _one / _other (no separate plural for 0 in simple nouns)
// French: singular for 0 and 1, plural for 2+ — use _one / _other
```

## Number / Date / Currency Formatting

**Always use `Intl` API** — never hardcode formats.

```typescript
// Numbers
new Intl.NumberFormat(locale).format(1234567)  // '1.234.567' (DE) vs '1,234,567' (EN)

// Dates
new Intl.DateTimeFormat(locale, { dateStyle: 'medium' }).format(new Date())

// Currency
new Intl.NumberFormat(locale, { style: 'currency', currency: 'EUR' }).format(9.99)
// '9,99 €' (DE/FR) vs '€9.99' (EN)
```

## Presentation Language (per session)

Sessions have an independent `presentationLanguage` field (≠ UI language):
- AI functions (recap, rephrasing, suggestions) use `presentationLanguage`
- `spellcheck` attribute on question input fields uses `presentationLanguage`
- Voter-facing strings use the voter's UI language (not `presentationLanguage`)

```typescript
// SessionConfig.tsx
<textarea
  lang={session.presentationLanguage}
  spellCheck={true}
/>
```

## Do Not
- Add string literals directly to JSX — always `t('key')` via `useTranslation`
- Use AI-generated translations without marking them for native review
- Hardcode locale-specific formats (numbers, dates, currency) — use `Intl`
- Add i18n to strings that are purely technical (error codes, log messages, API keys)
- Start i18n work before all R1–R3 strings are stable (per CLAUDE.md: i18n after Sprint 13)
